# AGENTS.md — Face Recognition Access Control System

## Project Overview

Face recognition door access system (人脸识别门禁系统) with user registration, real-time face scanning, liveness detection, and attendance tracking. **Design/planning repo** — implementation follows detailed plans in `docs/superpowers/plans/`.

**Scale**: <50 users, single access point, PC + USB camera.

## Build & Run Commands

### Backend (Python FastAPI)

```bash
cd backend
python -m venv venv && source venv/bin/activate
pip install -r requirements.txt
python -m app.init_db          # Initialize SQLite + seed config
python run.py                  # Start on http://localhost:8000
```

### Frontend (Vue 3 + Vite + TypeScript)

```bash
cd frontend
npm install
npm run dev                    # Dev server on http://localhost:5173
npm run build                  # Type-check (vue-tsc) + production build
npm run preview                # Preview production build
```

### Testing

```bash
# Backend — pytest (run from backend/)
pytest                              # All tests
pytest tests/test_face_utils.py     # Single test file
pytest tests/test_face_utils.py::TestFaceUtils::test_load_image_from_bytes  # Single test
pytest -v --tb=short                # Verbose with short tracebacks

# Frontend — Vitest + Playwright (run from frontend/)
npm run test                        # Vitest (watch mode)
npm run test:coverage               # Vitest with coverage report
npm run test:e2e                    # Playwright E2E tests
```

## Code Style — Backend (Python)

- **Naming**: `snake_case` for functions/variables, `PascalCase` for classes, `UPPER_SNAKE_CASE` for constants
- **Types**: Use type hints on all function signatures (`def register_face(user_id: int, image: np.ndarray) -> dict:`)
- **Imports**: stdlib → third-party → local, grouped with blank lines
- **Docstrings**: Chinese docstrings for public functions describing purpose, args, returns
- **Patterns**: Service layer (classes in `services/`), dependency injection via `Depends(get_db)`, singletons for AI services
- **API routers**: Use `APIRouter(prefix="/api/...", tags=[...])`, include via `app/api/__init__.py`
- **Response shape**: `{"code": 200, "data": {...}, "message": "..."}` — wrap in Pydantic response models
- **Error handling**: Raise `HTTPException(status_code=..., detail="...")`, never return raw error strings
- **Face reasons**: Use predefined reason strings: `no_face_detected`, `multiple_faces`, `poor_quality`, `liveness_failed`, `face_not_recognized`
- **File uploads**: Use `UploadFile = File(...)` for images, `multipart/form-data` content type
- **No type suppression**: Never use `# type: ignore` or `cast()` to silence type errors

## Code Style — Frontend (TypeScript/Vue)

- **Naming**: `camelCase` for variables/functions/composables, `PascalCase` for components/types/interfaces, `UPPER_SNAKE_CASE` for constants
- **Vue**: Always use Composition API with `<script setup lang="ts">`, never Options API
- **Pinia**: Use setup store syntax (`defineStore('name', () => { ... })`), not options syntax
- **Imports**: Use `@/` path alias for all `src/` imports (`import { useUserStore } from '@/stores'`)
- **API calls**: Centralized in `src/api/` modules, use typed generic helpers (`get<T>`, `post<T>`, `upload<T>`, `download`)
- **Error handling**: Axios interceptors handle HTTP errors globally; components catch business errors with try/catch + `ElMessage`
- **Types**: All API types in `src/types/`, import with `import type` syntax
- **No `any`**: Strict TypeScript (`strict: true`). Use proper interfaces, never `any`
- **Component naming**: Multi-word PascalCase (`FaceCapture.vue`, not `Facecapture.vue`)

## Key Conventions

- **Language**: Code in English, UI strings and comments in Chinese
- **Database**: SQLAlchemy ORM only, no raw SQL. Timestamps via `server_default=func.now()`
- **Feature vectors**: Stored as `.npy` files in `data/faces/encodings/`, loaded into memory at startup
- **Face images**: Stored as files in `data/faces/images/`, paths recorded in DB
- **Face threshold**: Default 0.6 confidence, configurable via `system_config` table
- **Liveness**: Blink detection (EAR algorithm) + head nod detection, requires 3-5 consecutive frames
- **WebSocket protocol**: JSON messages with `type` field (`"frame"`, `"result"`, `"status"`)
- **Image encoding**: Frontend sends base64-encoded JPEG at 0.8 quality via canvas

## Testing Guidelines

- **Backend fixtures**: Use `@pytest.fixture` for TestClient and DB sessions, clean up after each test
- **Test DB**: Use separate SQLite file for tests (`test_db.db`), drop/create tables per test run
- **Frontend mocking**: Mock `navigator.mediaDevices.getUserMedia` for camera tests
- **Test location**: Backend tests in `backend/tests/`, frontend in `frontend/tests/` and `frontend/e2e/`

## Common Pitfalls

- `dlib` requires CMake + C++ compiler on Linux/macOS, pre-built wheels on Windows
- SQLite `check_same_thread=False` is required for FastAPI multithreading
- WebRTC (getUserMedia) requires HTTPS in production; localhost is exempt
- Canvas `toDataURL` is synchronous; for large frames consider OffscreenCanvas
- Face recognition is CPU-bound — avoid running in async event loop; use `run_in_executor` if needed
